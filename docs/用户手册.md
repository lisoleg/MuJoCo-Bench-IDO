## 1. 项目简介

**MuJoCo-Bench-IDO** 是一个将 IDO/TOMAS 架构从离散符号域（ARC 拼图）迁移到连续物理控制域（MuJoCo 机器人）的基准验证平台。

### 什么是 IDO/TOMAS？

IDO（Information-Driven Orchestration）是一种**非冯诺依曼架构**的决策框架，核心思想是：

- **守恒优先**（Conservation-First）：先检查物理守恒定律是否被违反，再决定下一步行动
- **方向引导**（κ-Snap Gate）：通过 η（GaussEx 残差）衡量当前状态距离目标的远近，η 小到阈值以下才"扣合"执行
- **自演化**（ψ-Anchor Meta-Layer）：根据 η 趋势动态调整策略，像教练看比赛数据调整战术

TOMAS（Theory of Multi-Agent Systems）是 IDO 的理论基础，包含五层本体结构（L1-L5），从底层感知到顶层 Oracle 约束。

### MuJoCo-Bench-IDO 做什么？

它把 IDO 的决策循环映射到 MuJoCo 物理仿真环境，让机器人（人形、跳跃者、行走者、机械臂）在物理约束下尝试完成控制任务，同时验证 IDO 理论预言：

| 预言 | 内容 | 验证方式 |
|------|------|----------|
| P1 | IDO 永不违反物理守恒（NVR ≡ 0） | Noether-Check 门控 |
| P2 | IDO 比传统 RL 更快到达目标（SER ≥ 1.2） | 步数对比 |
| P3 | 传统 RL 会违反物理守恒（Baseline NVR > 0） | Baseline 评估 |

## 6. SIP-Bench 模式

### 什么是 SIP-Bench？

SIP-Bench（Self-evolving Iteration Protocol Benchmark）是 IDO 的纵向评估协议，测量 Agent 在**学习→适应→保持**三个阶段的性能变化。

### T0/T1/T2 三阶段

| 阶段 | 名称 | 说明 |
|------|------|------|
| **T0** | 初始基线 | 运行 IDO Agent，ψ-Anchor 观察但不主动调整 |
| **T1** | 迭代演化 | ψ-Anchor 主动演化（light/freeze 循环），调整 δ_K 和 Motor Primitive IC-Value |
| **T2** | 保持测试 | 使用 T1 演化后的策略，但不再进一步演化，测试改善是否持久 |

### 关键指标

- **Retention Gain** = T0_avg_steps / T2_avg_steps
  - > 1 → T2 比 T0 更快（改善持久）✅
  - = 1 → 无变化
  - < 1 → T2 更慢（改善未保持）

- **Stability Index** = T2_std_steps / T0_std_steps
  - < 1 → T2 比 T0 更稳定（方差降低）✅
  - = 1 → 稳定性相同
  - > 1 → T2 更不稳定

### 运行 SIP-Bench

```bash
# CLI 方式
python benchmarks/run_mujoco_bench.py --task humanoid-stand --eval-mode sip --episodes 5 --evolution-rounds 3

# Web 仪表盘方式
勾选 "SIP-Bench" toggle → 设置 evolution rounds → 点击 Run
```

## 7. 核心概念通俗解释

### η (GaussEx) — 距离目标的度量

> **通俗理解**: η 就像是你离家还有多远。越接近目标，η 越小，就像高斯分布——越靠近中心点密度越高。η 不是简单的距离，而是综合了位置、姿态、能量、速度四个维度的"加权距离平方"。

公式：$\eta = w_{vel} \cdot vel\_deficit + w_{height} \cdot height\_deficit^2 + w_{upright} \cdot upright\_deficit^2 + w_{eng} \cdot \max(0, E - E_{budget})^2$（locomotion η-mode）

> **注意**: v0.6.1 新增了 locomotion η-mode，用于 walker/cheetah/hopper 等行走任务。Point η-mode 用于 reacher/humanoid-stand 等定点任务。

### Noether 违规 — 物理守恒定律被违反

> **通俗理解**: Noether 违规就是"物理定律被违反的次数"。能量不该凭空产生或消失，力矩不该超过电机限制，机器人不该自己撞自己。IDO 的 Noether-Check 就像物理老师批改作业——每个动作都要通过物理定律考试。

三重守恒门 + 第四门（v0.6.0 新增）：
1. **能量门**: ΔE ≤ max_energy_inject + ε（能量不能凭空出现）
2. **力矩门**: |actuator_force| ≤ MAX_TORQUE × margin（不能超功率）
3. **碰撞门**: min_geom_distance ≥ threshold（不能自己撞自己）
4. **摩擦锥门** (v0.6.0): ||f_t|| ≤ μ·f_n（Coulomb 摩擦定律，μ=0.8）

### κ-Snap 门控 — 想清楚再行动

> **通俗理解**: κ-Snap 门控就像是"当优化足够收敛时才'扣合'执行"。还没想清楚就行动是危险的——η 太大说明还在探索，只有当 η 降到阈值以下（η < δ_K），才认为当前方向足够好，可以"扣合"到 PD 控制精细调整。

- η ≥ δ_K → 探索模式（Motor Primitive 尝试不同策略）
- η < δ_K → κ-Snap 模式（PD 稳定控制精细收敛）

### ψ-Anchor — 元管理层（教练角色）

> **通俗理解**: ψ-Anchor 就像教练看比赛数据调整战术。它监测 η 的趋势（上升、停滞、下降），动态调整 δ_K 阈值，决定是"放开探索"（light策略）还是"锁定当前最优"（freeze策略）。

- η 下降 → 收紧 δ_K（×0.8），追求更精确
- η 停滞 → 放宽 δ_K（×1.2），打破僵局
- η 上升 → 保持 δ_K 不变，避免恶化

### FlowMatching — 预判未来走势

> **通俗理解**: FlowMatching 就像天气预报——预测 η 的未来轨迹。如果预测 η 不会下降（停滞），就触发 ψ-Anchor 演化，换策略。

关键指标：
- **Hesitation-RMSE**: η 来回震荡的幅度（犹豫不决程度）
- **Retry-VOC**: η 方向频繁翻转的频率（反复尝试程度）

### epiplexity — 策略困惑度

> **通俗理解**: epiplexity 衡量策略的"困惑程度"。困惑度高说明当前策略不够用，需要演化新策略。就像考试前越焦虑，说明准备越不充分。

公式：$epiplexity = n_{invariants} \times (1/\delta_K) \times \log(max_{energy})$

### VG-Pair ≠ GAN — 不是对抗训练

> **通俗理解**: VG-Pair（Verifier-Generator Pair）和 GAN（生成对抗网络）看起来都有"生成器+验证器"，但本质完全不同。GAN 的验证器是**学出来的**，生成器试图骗过验证器——就像考试时学生和考官互相博弈。VG-Pair 的验证器是**物理定律本身**——你不可能骗过牛顿定律，能量守恒就是守恒。

| 对比 | VG-Pair | GAN |
|------|---------|-----|
| 目标 | 生成→验证→修正→执行 | min_G max_D 对抗博弈 |
| 验证器 | 物理定律（不可欺骗） | 学出来的判别器（可以骗） |
| Soundness | 物理定律保证 | 无形式化保证 |
| 训练方式 | 无对抗训练 | 对抗训练 |

### C-IPP — 连续交互式证明协议

> **通俗理解**: C-IPP 就像数学证明的交互式验证过程——提出一个解决方案→验证→如果验证失败就修正→再次验证→通过后执行。不是一次性提交答案，而是反复验证直到确定正确。

流程：**生成 → 验证 → 拒绝 → 修正 → 接受 → 执行**

### GEL — Goal-EML 注入损失

> **通俗理解**: GEL 把物理约束当作"惩罚项"，迫使策略遵守物理定律。就像给自动驾驶加上"不能闯红灯"的硬约束——不是靠学习后可能遵守，而是从一开始就被强制遵守。

公式：$L_{GEL} = \lambda_1 \cdot \|η_{Noether}\|^2 + \lambda_2 \cdot \|η_{contact}\|^2 + \lambda_3 \cdot \|η_{task}\|^2 + \lambda_4 \cdot hinge(task_{success})$

### Dual-Engine AGI — 双引擎驱动

> **通俗理解**: Dual-Engine AGI 用两个引擎驱动——数字引擎（LLM 推理，像大脑思考）+ 具身引擎（物理执行，像身体行动）。大脑负责规划和逻辑验证，身体负责物理世界验证和执行。两个引擎各自独立验证（VG-Pair），互相补充。

### 皮克定理(Pick's Theorem) — 离散几何先验

**一句话**：格点多边形面积 = 内部格点数 + 边界格点数/2 - 1，是离散版Gauss-Bonnet定理。

**通俗解释**：
想象一个画在方格纸上的多边形。皮克定理告诉你：只需数格子点就能算出面积！
- I = 多边形内部（不在边上）的格点数
- B = 多边形边界（在边上）的格点数
- 面积 = I + B/2 - 1

**IDO连接**：
- Noether-Check验证能量守恒 ΔE=0 → Pick-Check验证面积守恒 A=I+B/2-1
- 都是"离散不变量校验"：物理守恒 ↔ 几何守恒
- 预言P4：格点投影轨迹上 Pick残差≡0 ↔ Noether残差≡0
- 六方格点（60°密铺）是物理最优 → Hex-Nav benchmark设计依据

**加权皮克**：
- 内部格点权重 w_i=1，边界格点权重 w_b=1/2
- 这为ψ-Anchor的light/freeze策略提供了数学基础（分数权重替代二值阈值）

| 概念 | 物理层(IDO) | 几何层(Pick) |
|------|-----------|-------------|
| 守恒律 | ΔE = 0 (Noether) | A = I + B/2 - 1 (Pick) |
| 拓扑锚 | ψ-Anchor δ_K | χ(P) = 1 (Euler特征) |
| 最优格点 | (待验证) | 六方60°密铺 |

## 8. Benchmark 任务详解

### humanoid-stand: 人形机器人站立

- **自由度**: 21 DOF（3D 空间）
- **目标**: 人形机器人从初始姿态站立保持平衡
- **难点**: 多关节协调、重心控制、地面接触约束
- **GoalEML 不变量**: ee_at_target, torso_upright, no_self_collide
- **能量预算**: 500 J

### hopper-stand: 单腿跳跃机器人站立

- **自由度**: 6 DOF
- **目标**: 单腿机器人站立保持平衡
- **难点**: 单腿平衡、足部接触约束
- **GoalEML 不变量**: torso_upright, foot_on_ground, no_self_collide
- **能量预算**: 200 J

### walker-run: 双足行走机器人跑步

- **自由度**: 6 DOF
- **目标**: 双足机器人向前跑步不跌倒
- **难点**: 连续前进、动态平衡、跑步周期
- **GoalEML 不变量**: com_x_advancing, not_fallen, no_self_collide
- **能量预算**: 600 J

### reacher-easy: 2-DOF 机械臂到达目标

- **自由度**: 2 DOF（最简单）
- **目标**: 机械臂末端到达目标位置
- **难点**: 简单到达，适合入门调试
- **GoalEML 不变量**: ee_at_target
- **能量预算**: 50 J

### 7 项核心指标 Baseline 参考值

Dashboard 在每个指标卡片下方显示 baseline 参考值，帮助用户理解 IDO 与传统 RL（PPO/SAC）的差异。以下为 7 项核心指标的完整 baseline 参考（以 humanoid-stand 为例，各任务具体值见 `server.py` 中 `BASELINE_REFERENCE`）：

| 指标 | PPO 100k | SAC 100k | IDO 目标 | 说明 |
|------|----------|----------|----------|------|
| Episode Return | ~220 | ~180 | >300 | dm_control 标准尺度；IDO point η-mode 下 reward 尺度不同 |
| Success Rate | ~55% | ~48% | >70% | 任务成功率 |
| H_EML_residual | ~0.85 | — | <0.5 | 越低越好，η 接近目标 |
| Noether Violations (NVR) | ~800 | — | <100 | 1000步累计违规计数（碰撞检测为主，非比率） |
| Snap Efficiency | ~0.45 | — | >0.8 | Snap 触发比例 |
| Epiplexity | N/A | N/A | >300 | PPO/SAC 无 ψ-Anchor 不计算；IDO 目标表示足够结构信息支持演化 |
| CQ (Overall) | ~0.10 | ~0.12 | >0.80 | PPO/SAC 无良知框架 CQ 偏低；IDO 目标 >0.80 |

**关键说明**：

1. **NVR 计数方式**：NVR 是 1000 步累计违规计数（collision/torque/energy 各类违规之和），而非每 episode 比率或单步概率。高自由度任务（humanoid/hopper）碰撞违规较多，PPO 100k 后约 600-850 次；IDO 通过 Noether-Check 门控将目标降至 <100。Dashboard 中 NVR 显示的是累计值，baseline 参考也是累计值。

2. **Episode Return 尺度差异**：PPO/SAC 使用 dm_control 标准 reward 尺度（100k 步后 humanoid-stand ~220）。IDO 在 point η-mode 下 reward 尺度不同（基于 η-distance 而非原始物理 reward），累计 ~4-14 属正常现象，不可直接与 PPO/SAC 的数值做绝对量级对比。两者对比应关注趋势和成功率，而非绝对 return 值。

3. **Epiplexity 与 CQ 为 IDO 独有指标**：传统 RL（PPO/SAC）无 ψ-Anchor 元管理层和良知框架，因此这两个指标对 PPO/SAC 标记为 N/A。
   - **Epiplexity**（策略困惑度）：公式 $epiplexity = n_{invariants} \times (1/\delta_K) \times \log(max_{energy})$，值越高说明当前策略越"困惑"（约束多、精度要求高但表现不佳），触发 ψ-Anchor 演化新策略。IDO 目标 >200-300 表示系统积累了足够结构信息支持策略进化。
   - **CQ (Conscience Quotient)**（良知商数）：综合 Noether 守恒合规、PG-Gate 策略门控、Sentient 感知等多维度评分。PPO/SAC 无良知框架 CQ ~0.10-0.20；IDO 目标 >0.75-0.80。

## 11. 常见问题与故障排查

### mjviser 启动失败

**症状**: 点击 mjviser 按钮后无反应或报错

**排查步骤**:
1. 确认安装了 mjviser 和 viser：`pip install mjviser viser`
2. 确认 dm_control 正常：`python -c "import dm_control.suite; print('OK')"`
3. 查看终端输出的 `mjviser viewer failed:` 错误信息
4. 常见错误：`Viewer.__init__()` 参数不对 → v0.3.0 已修复此 bug

### 端口冲突

**症状**: 8080 或 8081 端口无法访问

**排查步骤**:
1. 8080 被 FastAPI 占用是正常的——检查是否有其他 FastAPI 服务占用
2. 8081 被 mjviser 占用是正常的——检查是否有其他 viser 服务占用
3. 查看占用进程：`netstat -ano | findstr 8080`（Windows）或 `lsof -i :8080`（Linux）
4. 杀死占用进程或更改端口配置

### dm_control 兼容性

**症状**: `import dm_control` 报错或环境加载失败

**排查步骤**:
1. 确认 Python 版本 ≥ 3.10
2. 安装 dm_control：`pip install dm_control`
3. 确认 mujoco 版本 ≥ 2.3.0：`pip show mujoco`
4. dm_control 需要 mujoco 作为底层依赖

### GPU 需求

**症状**: Cosmos-Predict baseline 无法运行

**排查步骤**:
1. Cosmos-Predict 7B/14B 模型需要 CUDA GPU
2. 检查 CUDA 是否可用：`python -c "import torch; print(torch.cuda.is_available())"`
3. 如无 GPU，跳过 Cosmos-Predict baseline（使用 `--ido_only` 参数）

### η 值异常大

**症状**: η 值超过 100 或 1000

**解释**: 这是正常的——不同任务的 η 尺度不同：
- reacher-easy: η ≈ 10000（机械臂距离远，η 自然大）
- walker-run: η ≈ 130（行走任务距离中等）
- humanoid-stand: η ≈ 2.6（站立任务距离近）
- hopper-stand: η ≈ 6.88（跳跃任务距离中等）

### Noether 违规数很大

**症状**: humanoid/hopper/walker 的 NV > 2000

**解释**: v0.2.2 数据中这三个任务确实存在大量 Noether 违规（NVR ≈ 1.47），原因：
1. Motor Primitive 设计未充分考虑碰撞约束
2. 能量预算参数需要调优
3. 只有 reacher-easy 实现了 NV ≡ 0（预言 P1 在此任务上通过）

**改善方向**: 使用 SIP-Bench 模式进行演化优化，或调整 GoalEML 参数。

##### 3.2.1 设计理念

基于"From VLA to Embodied Consciousness"（章锋, 2026）的架构设计，MuJoCo-Bench-IDO实现VLA模型对接框架。

**核心判词**：OpenVLA/Octo/π₀是P-Layer（现象意识层）的巨大进步——它们让机器人"会模仿"。但全员缺S-Layer（自我归因/κ-Snap）和C-Layer（ψ-锚硬约束）。MuJoCo-Bench-IDO的价值：往这些开源VLA模型上焊S-Layer + C-Layer + GaussEx η归约验证，使之从**具身智能**升格为**具身认知**。

##### 3.2.2 三层架构

| 层级 | 名称 | 功能 | 实现方式 |
|------|------|------|----------|
| **P-Layer** | 现象意识层 | 让机器人"会模仿" | VLA模型（OpenVLA/Octo/π₀）输出Joint Trajectory |
| **C-Layer** | ψ-锚硬约束层 | 物理约束检查、限幅/急停 | `PsiAnchorGate.check(action=q_raw)` → 限幅/急停 |
| **S-Layer** | 自我归因层 | 因果审计、自我监督 | 每步记录`KappaSnap(step, eta, tau_norm, violation)` |

**数据流**：
```
[VLA Backbone (OpenVLA/Octo/π₀)]
    ↓ Image + Language + Proprio → Joint Trajectory
[ψ-Anchor Gate (C-Layer)]
    ↓ 物理约束检查: MAX_TORQUE=0.050 N·m, MAX_VELOCITY, MAX_GRIP_FORCE
```
