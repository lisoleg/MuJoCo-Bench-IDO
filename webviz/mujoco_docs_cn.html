Introduction

MuJoCo 代表 Multi-Joint dynamics with Contact（多关节动力学与接触）。它是一个通用物理引擎，旨在促进机器人、生物力学、图形和动画、机器学习以及其他需要快速准确模拟铰接结构与其环境交互的领域的研究与开发。

它最初由 Roboti LLC 开发，于 2021 年 10 月被 Google DeepMind 收购并免费提供，并于 2022 年 5 月开源。MuJoCo 代码库可在 GitHub 上的 google-deepmind/mujoco 仓库中获取。

MuJoCo 是一个带有 C API 的 C/C++ 库，面向研究人员和开发人员。运行时模拟模块经过调优以最大化性能，并在由内置 XML 解析器和编译器预先分配的低级数据结构上运行。

用户使用原生 MJCF 场景描述语言（一种 XML 文件格式，设计为尽可能便于人类阅读和编辑）定义模型。也可以加载 URDF 模型文件。该库包括带有原生 GUI 的交互式可视化，在 OpenGL 中渲染。MuJoCo 还暴露了大量用于计算物理相关量的实用函数。

MuJoCo 可用于实现基于模型的计算，如控制合成、状态估计、系统辨识、机构设计、通过逆动力学进行数据分析，以及机器学习应用的并行采样。它也可以用作更传统的模拟器，包括用于游戏和交互式虚拟环境。

Core Features

MuJoCo 有很长的特性列表。在这里我们概述最显著的特性。

广义坐标结合现代接触动力学

物理引擎传统上分为两类。机器人学和生物力学引擎在广义或关节坐标中使用高效准确的递归算法。然而它们要么省略了接触动力学，要么依赖于早期的弹簧-阻尼器方法，这需要非常小的时间步长。游戏引擎使用更现代的方法，其中接触力通过求解优化问题来找到。然而，它们通常 resort 于过度指定的笛卡尔表示，其中关节约束被数值施加，当涉及复杂的运动学结构时会导致不准确和不稳定。

MuJoCo 开创了在广义坐标中进行模拟与基于优化的接触动力学的结合。这种方法此后已被其他模拟器采用。习惯于游戏引擎的用户最初可能会发现广义坐标违反直觉；请参阅下面的澄清部分。

柔软、凸且可解析求逆的接触动力学

在现代接触动力学方法中，由摩擦接触引起的力或冲量通常被定义为线性或非线性互补问题（LCP 或 NCP）的解，两者都是 NP-hard 的。MuJoCo 基于一种不同的接触物理公式，它简化为凸优化问题。我们的模型允许软接触和其他约束，并且具有唯一定义的逆，便于数据分析和控制应用。

默认的牛顿求解器提供二次收敛；替代算法包括共梯度法和可以处理椭圆摩擦锥的广义 Projected Gauss-Seidel 方法。求解器提供摩擦接触的统一处理，包括扭转和滚动摩擦、无摩擦接触、关节和肌腱限制、关节和肌腱中的干摩擦，以及各种等式约束。

腱几何

MuJoCo 可以建模腱的 3D 几何——即遵循包裹和经由点约束的最短路径长度字符串。该机制类似于 OpenSim 中的机制，但实现了一组更受限制的封闭形式包裹选项以加速计算。它还提供了机器人特有的结构，如滑轮和耦合自由度。腱可用于驱动以及施加对腱长度的不等式或等式约束。

通用驱动模型

在使用模型无关 API 的同时设计足够丰富的驱动模型是具有挑战性的。MuJoCo 通过采用抽象驱动模型实现了这一目标，该模型可以具有不同类型的传动、力生成和内部动力学（即使整体动力学成为三阶的状态变量）。这些组件可以被实例化，以统一的方式建模电机、气动和液压缸、PD 控制器、生物肌肉和许多其他驱动器。

可重新配置的计算流水线

MuJoCo 有一个顶层步进函数 mj_step，它运行整个前向动力学并推进模拟状态。然而，在模拟之外的许多应用中，能够运行计算流水线的选定部分是有益的。为此，MuJoCo 提供了大量标志，可以以任意组合设置，允许用户根据需要重新配置流水线。此外，许多较低级别的函数可以直接调用。用户定义的回调可以实现自定义力场、驱动器、碰撞例程和反馈控制器。

模型编译

如上所述，用户在称为 MJCF 的 XML 文件格式中定义 MuJoCo 模型。然后该模型由内置编译器编译成低级数据结构 mjModel，该结构经过交叉索引并针对运行时计算进行了优化。编译后的模型也可以保存在二进制 MJB 文件中。

模型与数据的分离

MuJoCo 在运行时将模拟参数分离为两个数据结构（C 结构体）：

  • mjModel 包含模型描述，预期保持不变。其中嵌入了包含模拟和可视化选项的其他结构，这些选项偶尔需要更改，但这是由用户完成的。
  • mjData 包含所有动态变量和中间结果。它用作一个草稿本，所有函数在那里读取输入并写入输出——然后这些输出成为模拟流水线中后续阶段的输入。它还包含一个预先分配并由内部管理的栈，因此运行时模块在模型初始化后不需要调用内存分配函数。

mjModel 由编译器构建。mjData 在运行时构建，给定 mjModel。这种分离使得模拟多个模型以及每个模型的多个状态和控制变得容易，进而促进了用于采样的多线程和有限差分。顶层 API 函数反映了这种基本分离，并具有以下格式：

void mj_step(const mjModel* m, mjData* d);

性能

MuJoCo 的运行时在初始化后执行零内存分配——所有工作内存都在 mjData 中预先分配。单个模拟步骤默认是单线程的，但约束岛启用了步骤内每个岛的并行性。大规模数据并行采样是通过创建共享同一 mjModel 的多个 mjData 实例并在不同线程上运行它们来实现的。

交互式模拟和可视化

原生 3D 可视化器提供网格和几何体的渲染、纹理、反射、阴影、雾、透明度、线框、天空盒、立体可视化（在支持四缓冲 OpenGL 的视频卡上）。此功能用于生成 3D 渲染，帮助用户深入了解物理模拟，包括自动生成的模型骨架、等效惯性盒、接触位置和法线、可以分离为法向和切向分量的接触力、外部扰动力、局部坐标系、关节和驱动器轴以及文本标签等视觉辅助。

可视化器期望一个带有 OpenGL 渲染上下文的通用窗口，从而允许用户采用他们选择的 GUI 库。随 MuJoCo 分发的代码示例 simulate.cc 展示了如何使用 GLFW 库做到这一点。

一个相关的可用性特性是能够"伸手进入"模拟，推动物体并查看物理如何响应。用户选择将施加外力和扭矩的物体，并看到扰动的实时渲染及其动态后果。这可用于可视化调试模型、测试反馈控制器的响应，或将模型配置到所需姿势。

强大而直观的建模语言

MuJoCo 有自己的建模语言叫 MJCF。MJCF 的目标是提供对所有 MuJoCo 计算能力的访问，同时使用户能够快速开发新模型并进行实验。这一目标在很大程度上是通过类似于 HTML 中内联层叠样式表（CSS）的广泛默认设置机制实现的。虽然 MJCF 有许多元素和属性，但用户在任何给定模型中需要设置的却惊人地少。这使得 MJCF 文件比许多其他格式更短、更可读。

复合柔性对象的自动生成

MuJoCo 的软约束可用于建模绳索、布料和可变形 3D 对象。这需要大量常规物体、关节、腱和约束协同工作。建模语言具有高级宏，由模型编译器自动扩展为标准模型元素的必要集合。重要的是，这些生成的柔性对象能够与模拟的其余部分完全交互。

约束岛和休眠

约束求解器自动发现独立的约束岛——由约束耦合的物体组，可以独立求解。这实现了每个岛的并行性并避免了过度求解：每个岛以其所需的迭代次数收敛，不受约束的自由度被完全跳过。此外，岛休眠可以冻结静止的岛，当场景的部分处于静止状态时，显著减少计算。

插件系统

MuJoCo 支持引擎插件，用自定义元素扩展物理引擎：传感器、驱动器和被动力。插件是动态加载的共享库，向引擎注册自身并按模型元素实例化。这允许在不修改核心引擎的情况下扩展 MuJoCo。

GPU 加速后端

除了 C 引擎，MuJoCo 还有用于大规模并行模拟的 GPU 加速后端：MJX（JAX）和 MuJoCo Warp（NVIDIA Warp）。两个后端消费相同的 mjModel 和 mjData 结构，允许在 CPU 和 GPU 之间无缝传输。一流的 Python 绑定提供对完整 C API 的访问。

Model Instances

在 MuJoCo 中有几个被称为"模型"的实体。用户在用 MJCF 或 URDF 编写的 XML 文件中定义模型。然后软件可以在不同的媒体（文件或内存）和不同的描述级别（高级或低级）中创建同一模型的多个实例。所有组合都是可能的，如下表所示：

高级低级
文件MJCF/URDF (XML)MJB (binary)
内存mjSpec (C struct)mjModel (C struct)

所有运行时计算都使用 mjModel 执行，它太复杂而无法手动创建。这就是为什么我们有两个建模级别。高级别是为了用户方便：其唯一目的是被编译成可以在其上执行计算的低级别模型。生成的 mjModel 可以加载并保存到二进制文件（MJB），然而这些是版本特定的且无法反编译，因此模型应始终作为 XML 文件维护。

mjSpec C 结构体与 MJCF 文件格式一一对应。XML 加载器解释 MJCF 或 URDF 文件，创建相应的 mjSpec 并将其编译为 mjModel。用户可以编程方式创建 mjSpec，然后将其保存为 MJCF 或编译它。程序化模型创建和编辑在模型编辑章节中描述。

以下图表显示了获得 mjModel 的不同路径：

  • (文本编辑器) → MJCF/URDF 文件 → (MuJoCo 解析器 → mjSpec → 编译器) → mjModel
  • (用户代码) → mjSpec → (MuJoCo 编译器) → mjModel
  • MJB 文件 → (模型加载器) → mjModel

Examples

以下是 MuJoCo MJCF 格式中的一个简单模型。它定义了一个固定在世界上的平面、一个更好地照亮物体并投射阴影的灯光，以及一个具有 6 DOF 的浮动盒子（这就是"自由"关节所做的）。

hello.xml：

<mujoco>
  <worldbody>
    <light diffuse=".5 .5 .5" pos="0 0 3" dir="0 0 -1"/>
    <geom type="plane" size="1 1 0.1" rgba=".9 0 0 1"/>
    <body pos="0 0 1">
      <joint type="free"/>
      <geom type="box" size=".1 .2 .3" rgba="0 .9 0 1"/>
    </body>
  </worldbody>
</mujoco>

如果模拟此模型，盒子将落到地面上。下面给出了无渲染的被动动力学基本模拟代码。

#include "mujoco.h"
#include "stdio.h"

char error[1000];
mjModel* m;
mjData* d;

int main(void) {
  // 从文件加载模型并检查错误
  m = mj_loadXML("hello.xml", NULL, error, 1000);
  if (!m) {
    printf("%s\n", error);
    return 1;
  }

  // 创建对应模型的数据
  d = mj_makeData(m);

  // 运行模拟 10 秒
  while (d->time < 10)
    mj_step(m, d);

  // 释放模型和数据
  mj_deleteData(d);
  mj_deleteModel(m);

  return 0;
}

这技术上是一个 C 文件，但它也是一个合法的 C++ 文件。实际上 MuJoCo API 与 C 和 C++ 都兼容。通常用户代码会用 C++ 编写，因为它增加了便利性，并且不会牺牲效率，因为计算瓶颈在于已经高度优化的模拟器中。

函数 mj_step 是推进模拟状态一个时间步的顶层函数。这个例子当然只是一个被动动力学系统。当用户指定控制或施加力并开始与系统交互时，事情变得更加有趣。

接下来我们提供一个更详细的示例，说明 MJCF 的几个特性。考虑以下 example.xml：

<mujoco model="example">
  <default>
    <geom rgba=".8 .6 .4 1"/>
  </default>

  <asset>
    <texture type="skybox" builtin="gradient" rgb1="1 1 1" rgb2=".6 .8 1" width="256" height="256"/>
  </asset>

  <worldbody>
    <light pos="0 1 1" dir="0 -1 -1" diffuse="1 1 1"/>
    <body pos="0 0 1">
      <joint type="ball"/>
      <geom type="capsule" size="0.06" fromto="0 0 0  0 0 -.4"/>
      <body pos="0 0 -0.4">
        <joint axis="0 1 0"/>
        <joint axis="1 0 0"/>
        <geom type="capsule" size="0.04" fromto="0 0 0  .3 0 0"/>
        <body pos=".3 0 0">
          <joint axis="0 1 0"/>
          <joint axis="0 0 1"/>
          <geom pos=".1 0 0" size="0.1 0.08 0.02" type="ellipsoid"/>
          <site name="end1" pos="0.2 0 0" size="0.01"/>
        </body>
      </body>
    </body>

    <body pos="0.3 0 0.1">
      <joint type="free"/>
      <geom size="0.07 0.1" type="cylinder"/>
      <site name="end2" pos="0 0 0.1" size="0.01"/>
    </body>
  </worldbody>

  <tendon>
    <spatial limited="true" range="0 0.6" width="0.005">
      <site site="end1"/>
      <site site="end2"/>
    </spatial>
  </tendon>
</mujoco>

此模型是一个 7 自由度臂"握住"一根绳子，另一端连接一个圆柱体。绳子实现为具有长度限制的腱。肩部有球关节，肘部和腕部有成对的铰链关节。圆柱体内的盒子表示一个自由"关节"。XML 中最外层的 body 元素是必需的 worldbody。请注意，在两个物体之间使用多个关节不需要创建虚拟物体。

MJCF 文件包含指定模型所需的最少信息。胶囊由空间中的线段定义——在这种情况下只需要胶囊的半径。物体坐标系的位置和方向从属于它们的几何体推断出来。惯性属性在均匀密度假设下从 geom 形状推断。两个 site 被命名是因为腱定义需要引用它们，但其他东西都没有命名。关节轴仅为铰链关节定义，而不是球关节。碰撞规则自动定义。摩擦属性、重力、模拟时间步等设置为默认值。顶部指定的默认 geom 颜色适用于所有 geom。

除了将编译后的模型保存在二进制 MJB 格式中，我们还可以将其保存为 MJCF 或人类可读文本。XML 版本与原始版本相似，而文本版本包含 mjModel 中的所有信息。将文本版本与 XML 版本进行比较，揭示了模型编译器为我们做了多少工作。

Model Elements

本节简要描述可以包含在 MuJoCo 模型中的所有元素。

选项

每个模型有三组选项。它们总是被包含。如果它们的值未在 XML 文件中指定，则使用默认值。选项的设计使得用户可以在每个模拟时间步之前更改它们的值。然而在一个时间步内，不应更改任何选项。

mjOption

此结构包含影响物理模拟的所有选项。它用于选择算法并设置其参数、启用和禁用模拟流水线的不同部分，以及调整系统级物理属性如重力。

mjVisual

此结构包含所有可视化选项。还有额外的 OpenGL 渲染选项，但这些是会话相关的，不属于模型的一部分。

mjStatistic

此结构包含由编译器计算的关于模型的统计信息：平均物体质量、模型的空间范围等。它是为了信息目的而包含的，也因为可视化器使用它进行自动缩放。

资产

资产本身不是模型元素。模型元素可以引用它们，在这种情况下，资产以某种方式改变引用元素的属性。一个资产可以被多个模型元素引用。由于包含资产的唯一目的是引用它，而引用只能通过名称完成，所以每个资产都有一个名称（适用时可以从文件名推断）。相比之下，常规元素的名称可以留为未定义。

网格 (Mesh)

MuJoCo 可以从 OBJ 文件和二进制 STL 加载三角化网格。虽然任何三角形集合都可以加载并可视化为网格，但碰撞检测器使用凸包工作。有编译时选项用于缩放网格，以及将基本几何形状拟合到它。网格还可以用于自动推断惯性属性——通过将其视为三角形金字塔的联合并组合它们的质量和惯性。请注意，网格没有颜色，而是使用引用 geom 的材质属性为网格着色。

皮肤 (Skin)

蒙皮网格（或皮肤）是其形状可以在运行时变形的网格。它们的顶点附着到刚体（在此上下文中称为骨骼），每个顶点可以属于多个骨骼，导致皮肤的平滑变形。皮肤纯粹是可视化对象，不影响物理，但它们可以显著增强视觉真实感。

高度场 (HeightField)

高度场可以从 PNG 文件或自定义二进制格式的文件加载。高度场是高程数据的矩形网格。编译器将数据归一化到范围 [0-1]。高度场的实际空间范围然后由引用 geom 的尺寸参数确定。高度场只能从附着到世界体的 geom 引用。为了渲染和碰撞检测目的，网格矩形被自动三角化。与这种复合对象的碰撞检测原则上可以为单个 geom 对生成大量接触点。如果发生这种情况，只保留前 64 个接触点。

纹理 (Texture)

纹理可以从 PNG 文件加载或由编译器基于用户定义的程序参数合成。还有在模型创建时将纹理留空并在运行时稍后更改的选项。可视化器支持两种类型的纹理映射：2D 和立方体。2D 映射对于平面和高度场有用。立方体映射对于无需指定纹理坐标即可将纹理"收缩包裹"在 3D 对象周围有用。与所有其他直接从模型元素引用的资产不同，纹理只能从另一个资产（即材质）引用。

材质 (Material)

材质用于控制 geom、site 和腱的外观。外观包括纹理映射以及与 OpenGL 灯光交互的其他属性：RGBA、镜面反射度、光泽度、发光度。材质还可以用于使对象具有反射性。请注意，模型元素也可以有局部 RGBA 参数。如果材质和局部 RGBA 都指定了，局部定义具有优先权。

运动学树

MuJoCo 模拟通常受约束的刚体集合的动力学。系统状态在关节坐标中表示，物体被显式组织成运动学树。树结构由 mjModel.body_parentid 给出。顶层"世界"体始终存在（id 为 0）并且是其自身的父体。不允许运动学环；如果需要环关节，应该用等式约束建模。

体 (Body)

体具有质量和惯性属性，但没有任何几何属性。相反，几何形状（或 geom）附着到体上。每个体有两个坐标系：用于定义它以及定位其他元素相对于它的坐标系，以及以体质心为中心并与其惯性主轴对齐的惯性坐标系。在每个时间步，MuJoCo 递归地计算前向运动学，产生所有体在全局笛卡尔坐标中的位置和方向。

关节 (Joint)

关节在体内定义。它们在体和其父体之间创建运动自由度（DOF）。在没有关节的情况下，体被焊接到其父体上。有四种类型的关节：ball、slide、hinge 和"自由关节"，后者创建浮动体。单个体可以有多个关节。这样复合关节被自动创建，无需定义虚拟体。

关节参考

参考姿势是存储在 mjModel.qpos0 中的关节位置向量。它对应于模型处于其初始配置时关节的数值。所有关节的参考值被汇编到向量 mjModel.qpos0 中。每当模拟重置时，关节配置 mjData.qpos 被设置为 mjModel.qpos0。在运行时，关节位置向量被解释为相对于参考姿势。此变换是附加到存储在 mjModel 的体元素中的父子平移和旋转偏移之上的。ref 属性仅适用于标量关节（slide 和 hinge）。对于球关节，保存在 mjModel.qpos0 中的四元数始终为 (1,0,0,0)。

弹簧参考

这是所有关节和腱弹簧达到其静止长度的姿势。当关节配置偏离弹簧参考姿势时产生弹簧力，且与偏离量成线性关系。弹簧参考姿势保存在 mjModel.qpos_spring 中。

DOF

自由度与关节密切相关，但不是一一对应的关系，因为球关节和自由关节有多个 DOF。将关节视为指定位置信息，将 DOF 视为指定速度和力信息。DOF 具有与速度相关的属性，如摩擦损失、阻尼、电枢惯性。DOF 不是由用户直接指定的，而是由编译器根据关节创建。

树 (Tree)

运动学树或"树"是一个可移动的体及其所有后代。因此世界和其他静态体在全局树结构中，但不与任何树关联。因为全局树结构使用深度优先组织，属于单个树的所有体、关节和 DOF 始终是连续的。树的数量由 mjModel.ntree 给出。

几何体 (Geom)

Geom 是刚性地附着到体的 3D 形状。多个 geom 可以附着到同一个体。这在考虑到 MuJoCo 仅支持凸 geom-geom 碰撞的情况下特别有用，创建非凸对象的唯一方法是将它们表示为凸 geom 的联合。除了碰撞检测和接触力的计算，geom 还用于渲染，以及自动推断体质量和惯性。

站点 (Site)

Site 本质上是轻量级 geom。它们不参与碰撞检测或惯性属性的自动计算，但它们可以用于指定其他对象的空间属性，如传感器、腱路由和滑块曲柄端点。

相机 (Camera)

可以在模型中定义多个相机。始终有一个默认相机，用户可以在交互式可视化器中用鼠标自由移动。然而定义额外的相机通常很方便，它们要么固定在世界中，要么附着到其中一个体并随之移动。

灯光 (Light)

灯光可以固定到世界体或附着到运动体上。可视化器提供对 OpenGL 中完整照明模型的访问，包括环境光、漫反射和镜面反射分量、衰减和截止、位置和方向照明、雾。灯光也可以投射阴影。

独立元素

这里我们描述不属于单个体的模型元素，因此在运动学树之外描述。

腱 (Tendon)

腱是标量长度元素，可用于驱动、施加限制和等式约束，或创建弹簧-阻尼器和摩擦损失。有两种类型的腱：固定腱和空间腱。固定腱是（标量）关节位置的线性组合，用于建模机械耦合。空间腱被定义为通过一系列指定 site 或包裹指定 geom 的最短路径。空间腱也可以使用滑轮分成多个分支。

驱动器 (Actuator)

MuJoCo 提供灵活的驱动器模型，具有三个可以独立指定的组件：传动、激活动力学和力生成。传动指定驱动器如何附着到系统的其余部分；可用类型是关节、腱、滑块曲柄、site 和体。激活动动力学可用于建模气动或液压缸以及生物肌肉的内部激活状态。力生成机制确定控制信号如何映射到标量力。

传感器 (Sensor)

MuJoCo 可以生成保存在全局数组 mjData.sensordata 中的模拟传感器数据。结果不用于任何内部计算；而是提供给用户。可用的传感器类型包括触摸传感器、惯性测量单元（IMU）、力-扭矩传感器、关节和腱位置和速度传感器、驱动器传感器、体坐标系传感器、测距仪和许多其他。

等式约束 (Equality Constraint)

等式约束可以在运动学树结构和其中定义的关节/DOF 已经施加的约束之外施加额外的约束。它们可用于创建环关节，或一般地建模机械耦合。可用的等式约束类型有：在一个点连接两个体、将两个体焊接在一起、固定关节或腱的位置、通过三次多项式耦合两个关节或两个腱的位置、约束 flex 的边缘到其初始长度。

Flex

Flex 表示可以是 1、2 或 3 维的可变形网格（因此它们的元素是胶囊、三角形或四面体）。与刚性地附着到单个体的静态形状的 geom 不同，flex 的元素是可变形的：它们通过连接多个体来构造。这些可变形元素支持碰撞和接触力，以及产生被动力和约束力，软性地保持可变形实体的形状。

接触对 (Contact Pair)

被检查接触的 geom 对可以来自两个来源：自动邻近测试和过滤器，以及模型中提供的 geom 对的显式列表。后者是一种单独的模型元素类型。显式指定允许用户以动态机制无法做到的方式定义接触参数。

接触排除 (Contact Exclude)

与接触对相反：它指定应该从候选接触对生成中排除的体对。它对于禁用由于几何形状导致不必要永久接触的体之间的接触有用。

自定义数值 (Custom Numeric)

有三种方式在 MuJoCo 模拟中输入自定义数值：全局数值字段、元素特定的自定义数组，以及 mjData.userdata 数组（不被任何 MuJoCo 计算使用，用户可以存储自定义计算的结果）。

自定义文本 (Custom Text)

自定义文本字段可以保存在模型中。它们可用于自定义计算——要么指定关键字命令，要么提供一些其他文本信息。但不要用它们做注释；在编译模型中保存注释没有好处。

自定义元组 (Custom Tuple)

自定义元组是 MuJoCo 模型元素的列表，可能包括其他元组。它们不被模拟器使用，但可用于指定用户代码需要的元素组。

关键帧 (Keyframe)

关键帧是模拟状态变量的快照。它包含关节位置、关节速度、驱动器激活、控制、mocap 体姿势和模拟时间。模型可以包含关键帧库。它们对于将系统状态重置到感兴趣的点有用。请注意，关键帧不是用于在模型中存储轨迹数据的。

Clarifications

读者可能对其他物理模拟器和相关约定有经验。本节的目标是 preemptively 澄清最可能令人混淆的方面。

仿真发散

模拟的发散发生在状态元素快速趋向无穷大时。在 MuJoCo 中，这通常表现为 mjWARN_BADQACC 警告。发散在所有物理模拟中都是固有的，不一定表明模型不好或模拟器有 bug，而是提示对于给定的积分器选择，时间步太大。

在物理模拟中，速度（大时间步）和稳定性（小时间步）之间始终存在张力。应该可以通过减小时间步和/或切换到更稳定的积分器来防止发散。如果那失败了，原因就不同了——例如在物体初始化在穿透中的模型中，大的排斥力可能将它们推开并导致发散。

单位未指定

MuJoCo 不指定基本物理单位。用户可以选择他们想要的单位系统解释，只要它是一致的（MKS 或 CGS 都是一致的单位系统）。

鼓励用户使用 MKS，因为有两个地方 MuJoCo 使用类似 MKS 的默认值：

  • 重力的默认值为 (0, 0, -9.81)，对应于 MKS 中的地球表面重力。
  • geom 密度（用于推断体质量和惯性）的默认值为 1000，对应于 MKS 中水的密度。

一旦选择了一致的基本单位系统，所有导出单位都对应此系统。例如，如果模型被解释为 MKS，则力和扭矩分别为牛顿和牛顿-米。

角度：虽然在 MJCF 中可以使用度指定角度（度是默认的），但 mjModel 和 mjData 中的所有角量以弧度表示。

意外碰撞

MuJoCo 默认排除具有直接父子关系的体对之间的 geom 碰撞。例如，在臂模型中：即使胶囊 geom 穿透，在"肘部"也没有碰撞，因为前臂是上臂的直系子体。

然而，此排除不应用于父体是静态体的情况（世界体，或相对于世界体没有任何自由度的体）。这防止物体穿过地板或穿过墙壁。然而此行为经常导致：用户注释掉浮动基模型的根关节以防止它掉落；现在基体被计为静态，出现了以前不存在的新碰撞。

有两种简单的方法来避免此问题：

  1. 不要移除根关节。也许禁用重力并可能添加一些流体粘度以防止模型移动太多就足够了。
  2. 使用碰撞过滤显式禁用不需要的碰撞，要么通过设置相关的 contype 和 conaffinity 属性，要么通过使用接触排除指令。

数据导向设计

MuJoCo 使用数据导向架构而不是面向对象架构。代替对象，我们有少量数据结构和大量操作它们的函数。

我们将模型（mjModel）与数据（mjData）分离。所有顶层函数期望指向 mjModel 和 mjData 的指针作为参数。此设计避免全局变量并自然支持多线程：每个线程简单地在其自己的 mjData 实例上操作。

柔性与滑动

MuJoCo 基于接触和其他约束物理的数学模型。此模型本质上是软的——更用力地推约束将始终导致更大的加速度，因此逆动力学可以被唯一定义。这产生凸优化问题并启用依赖逆动力学的分析。

此软模型的一个经常令人混淆的方面是渐进接触滑动无法避免。类似地，摩擦关节将在重力下逐渐屈服。这不是因为求解器无法防止滑动，而是因为它首先就没有试图防止滑动。如果滑动要被完全抑制，"更用力地推→更大加速度"的关键属性就必须被违反。

所以如果你在模拟中看到渐进滑动，需要调整 solref 和 solimp 参数向量来减少此效果。增加约束阻抗以及全局 mjModel.opt.impratio 设置可以特别有效。

对于希望完全抑制滑动的情况，有一个 NoSlip 后处理求解器，在主求解器之后运行。然而当使用此选项时，MuJoCo 不再解决它被设计来解决的凸优化问题，模拟可能变得不那么鲁棒。因此使用牛顿求解器配合椭圆摩擦锥和大的 impratio 值是减少滑动的推荐方式。

类型、名称、ID

可用的元素类型在 mjtype.h 中定义，在枚举类型 mjtObj 中。MuJoCo API 中的函数 mj_name2id 和 mj_id2name，将元素名称映射到整数 ID，反之亦然。

在 XML 中命名模型元素是可选的。相同类型的两个元素不能有相同的名称。只有在给定元素需要在模型的其他地方被引用时才需要命名。一旦模型被编译，名称仍然存储在 mjModel 中为了用户方便。

元素的整数 ID 对于索引 MuJoCo 数据数组是必不可少的。ID 是0 基的，遵循 C 约定。

int jntid = mj_name2id(m, mjOBJ_JOINT, "elbow");
if (jntid >= 0)
   printf("(%f, %f)\n", m->jnt_range[2*jntid], m->jnt_range[2*jntid+1]);

如果名称未找到，函数返回 -1，这就是为什么应该始终检查 id>=0。

体、几何体、站点

体、geom 和 site 是 MuJoCo 元素，大致对应于物理世界中的刚体。那为什么它们是分开的？

相似之处：体、geom 和 site 都有附着到它们的空间坐标系。这些坐标系的位置和方向在每个时间步从 mjData.qpos 通过前向运动学计算。

区别：

  • 体：用于构造运动学树并且是其他元素（包括 geom 和 site）的容器。体有空间坐标系、惯性属性，但没有与外观或碰撞几何相关的属性。
  • Geom：用于指定外观和碰撞几何。每个 geom 属于一个体并刚性地附着到该体。多个 geom 可以附着到同一个体。一个 geom 还可以有密度或质量值，模型编译器使用它们来计算父体的质量和惯性。
  • Site：轻量级 geom。它们有相同的外观属性但不能参与碰撞，也不能用于推断体质量。Site 可以指定触摸传感器的体积、IMU 传感器的附着、空间腱的路由、滑块曲柄驱动器的端点。

以下示例说明了多个 site 和 geom 可以附着到同一个体：

<mujoco>
  <worldbody>
    <body pos="0 0 0">
      <geom type="sphere" size=".1" rgba=".9 .9 .1 1"/>
      <geom type="capsule" pos="0 0 .1" size=".05 .1" rgba=".9 .9 .1 1"/>
      <site type="box" pos="0 -.1 .3" size=".02 .02 .02" rgba=".9 .1 .9 1"/>
      <site type="ellipsoid" pos="0 .1 .3" size=".02 .03 .04" rgba=".9 .1 .9 1"/>
    </body>
  </worldbody>
</mujoco>

关节坐标

MuJoCo 和游戏引擎之间的关键区别之一是 MuJoCo 在广义或关节坐标中操作，而大多数游戏引擎在笛卡尔坐标中操作。两种方法之间的差异：

关节坐标笛卡尔坐标
最适合复杂的运动学结构，如机器人最适合相互弹跳的许多体
关节在默认焊接的体之间添加自由度关节在默认自由浮动的体之间移除自由度
关节约束在表示中是隐式的，不能被违反关节约束被数值强制执行，可以被违反
位置和方向通过前向运动学获得，不能直接操纵位置和方向被显式表示，可以直接操纵

浮动物体

在关节坐标中工作时，你不能简单地将任意体的位置和方向设置为你想要的任何值。对于浮动体（通过自由关节连接到世界的体），情况不同。此类体的位置和方向以及线性和角速度在 mjData.qpos 和 mjData.qvel 中被显式表示，因此可以直接操纵。

自由关节的语义如下：

  • 位置数据是 7 个数字（3D 位置后跟单位四元数）
  • 速度数据是 6 个数字（3D 线性速度后跟 3D 角速度）
  • 自由关节的线性位置和线性速度在全局坐标系中
  • 自由关节的方向（四元数）也在全局坐标系中
  • 自由关节的旋转速度在局部体坐标系中
